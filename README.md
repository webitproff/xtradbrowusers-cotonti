## Что это за плагин и зачем он нужен?

Extrafields Users Custom — это специализированное расширение для Cotonti CMF, которое позволяет добавлять неограниченное количество произвольных полей (extrafields) к профилям пользователей, **не изменяя при этом структуру стандартной таблицы `cot_users`**. Все дополнительные данные сохраняются в отдельной, изолированной таблице `cot_xtradbrowusers`, связанной с основной записью пользователя через внешний ключ с каскадным удалением (`ON DELETE CASCADE`).

Такой подход особенно актуален для сайтов, где профиль пользователя — это не просто учётная запись, а лицо в сообществе, витрина мастера или карточка клиента. Будь то форум по интересам, интернет-магазин, фриланс-биржа или корпоративный портал — плагин позволяет легко адаптировать профиль под любой сценарий, не вмешиваясь в ядро системы.

Ключевые преимущества изолированного хранения:

- **Чистота ядра системы:** стандартная таблица пользователей остаётся нетронутой, что исключает конфликты при обновлениях Cotonti и упрощает обслуживание.
- **Полная изоляция данных:** любые изменения структуры дополнительных полей не затрагивают основные данные пользователя.
- **Автоматическая целостность:** при удалении пользователя все связанные дополнительные данные удаляются автоматически (благодаря внешнему ключу), а также очищаются загруженные файлы.
- **Простота отката:** при деинсталляции плагина все его данные полностью удаляются одним SQL‑запросом и очисткой папок.

## Что идёт «из коробки»?

Сразу после установки плагина администратор получает **15 заранее настроенных демонстрационных полей** всех основных типов, поддерживаемых API Extrafields в Cotonti. Среди них:

- Текстовые поля (input, textarea)
- Целые числа и числа с плавающей точкой (inputint, double)
- Выпадающие списки (select)
- Радиокнопки (radio)
- Дата и время (datetime)
- Флаг страны (country)
- Диапазон чисел (range)
- Чекбоксы с множественным выбором (checklistbox)
- Загрузка файлов (file)

Это позволяет не только сразу увидеть плагин в действии, но и быстро оценить, какой тип поля лучше подходит для конкретной задачи — без долгого изучения документации.

## Кому и почему будет полезен этот плагин?

Плагин решает задачу расширения профиля пользователя там, где стандартного набора полей недостаточно. В зависимости от типа проекта вы получаете возможность вывести на первый план именно те данные, которые важны вашей аудитории:

**Для сообществ и форумов:**
- Показать статус, достижения, репутацию, ссылки на соцсети.
- Добавить поля с интересами или специализацией, чтобы участники находили единомышленников.
- Дать возможность кратко рассказать о себе, загрузить аватар в отдельное поле.

**Для интернет-магазинов:**
- Хранить дополнительный телефон, предпочтительный способ связи, адрес доставки (если не используется отдельная адресная книга).
- Вести учёт клиентов по категориям (оптовый/розничный, уровень лояльности).
- Прикреплять к профилю документы: договоры, реквизиты.

**Для фриланс-бирж и площадок услуг:**
- Публиковать портфолио, список навыков, сертификаты.
- Указывать профессиональный опыт, ставку, график работы.
- Загружать резюме прямо в профиль.
- Выводить рейтинг исполнителя, верификацию, уровень квалификации — всё это можно оформить в виде отдельных полей и гибко кастомизировать вывод.

Разработчикам и администраторам плагин даёт полную свободу: все поля управляются через стандартный интерфейс (`Администрирование → Расширения → Экстраполя`), а их значения автоматически доступны в шаблонах регистрации, редактирования профиля (пользователем и администратором), публичной карточки и общего списка пользователей. Также поддерживается вывод в мета‑теги страницы (`header.tpl`) через специальный хук, что открывает дополнительные возможности для SEO.

Благодаря полной интеграции с системой тегов Cotonti каждое поле можно выводить как в универсальном цикле, так и индивидуально, с собственной вёрсткой и логикой отображения. Это делает плагин одинаково удобным как для быстрого добавления пары полей, так и для глубокой кастомизации профилей на крупных проектах.

## Структура и назначение файлов плагина

Плагин Extrafields Users Custom взаимодействует с Cotonti через систему хуков. Каждый файл, предназначенный для выполнения на определённом хуке, содержит обязательный заголовок, который регистрирует его в таблице `cot_plugins` и определяет, когда именно код должен быть выполнен. Например, файл, подключающийся к хуку `admin.extrafields.first`, начинается так:

```php
<?php
/* ====================
  [BEGIN_COT_EXT]
  Hooks=admin.extrafields.first
  [END_COT_EXT]
==================== */
```

После установки плагина эта запись попадает в таблицу `cot_plugins`, и при каждом вызове `cot_getextplugins('admin.extrafields.first')` Cotonti автоматически подключает и выполняет наш файл.

### Группировка файлов

#### Основные файлы плагина


### 1. Файлы регистрации и инициализации плагина

#### Файл `xtradbrowusers.setup.php`

Это **файл метаданных плагина**, который не является обработчиком какого‑либо хука. Он содержит единственный блок `[BEGIN_COT_EXT]...[END_COT_EXT]`, в котором через пары `Ключ=Значение` объявлены основные сведения о расширении:

- `Code` – уникальный идентификатор плагина (`xtradbrowusers`);
- `Name` – человекочитаемое название (`Custom Extrafields Users`);
- `Description`, `Version`, `Date`, `Author`, `Copyright`, `Notes` – справочная информация;
- `Requires_modules` – список обязательных модулей (в нашем случае `users`);
- `Auth_guests`, `Lock_guests`, `Auth_members`, `Lock_members` – права доступа для гостей и участников, которые будут записаны в таблицу `cot_auth` при установке.

**Этот файл не содержит PHP‑кода, исполняемого при каждом запросе.** Он обрабатывается **единожды** функцией `cot_extension_install()` во время установки плагина. При этом система Cotonti автоматически:

- создаёт запись в таблице `cot_core` с `ct_code='xtradbrowusers'`, `ct_plug=1` (признак плагина), текущей версией;
- на основе значений `Auth_guests`, `Lock_guests`, `Auth_members`, `Lock_members` формирует строки в таблице `cot_auth` для каждой группы пользователей (гости, неактивные, участники, суперадминистраторы и т.д.);
- сканирует все файлы плагина, находит в них заголовки вида `Hooks=...`, и для каждого создаёт отдельную запись в таблице `cot_plugins` с указанием хука, части, порядка выполнения и пути к файлу.

Сам `setup.php` **не вызывает** `Cot::$db->registerTable('xtradbrowusers')` и не подключает языковые файлы — эта инициализация выполняется позже, в рабочих файлах плагина.  

Файл не добавляет записей в `cot_config`, так как плагин пока не имеет собственных настроек, однако при необходимости разработчик может расширить `setup.php` блоком `COT_EXT_CONFIG`, и система автоматически создаст нужные опции в таблице `cot_config` (структура `cot_config` полностью позволяет хранить любые параметры модулей и плагинов).

#### Файл `xtradbrowusers.global.php`

Этот файл уже содержит **активный хук** `global`. Благодаря записи в `cot_plugins` он подключается **при каждом запросе** к сайту (до формирования основного вывода). Его задача — гарантировать, что во всех точках выполнения Cotonti доступны:

- языковые константы плагина (через `cot_langfile('xtradbrowusers', 'plug')`);
- API‑функции плагина, определённые в `inc/xtradbrowusers.functions.php`.

Именно внутри `xtradbrowusers.functions.php` (подключаемого отсюда) находится вызов `Cot::$db->registerTable('xtradbrowusers')`, регистрирующий глобальную переменную `$db_xtradbrowusers` и делающий доступным обращение `Cot::$db->xtradbrowusers` во всём коде плагина и за его пределами.  

Таким образом, `global.php` обеспечивает «фоновую» инициализацию, не требующую повторения в каждом хуке, а `setup.php` выполняет разовую регистрацию плагина в системных таблицах при установке.

#### 2. Файлы установки и удаления (`setup/`)

- **`setup/xtradbrowusers.install.sql`**  
  Создаёт таблицу `cot_xtradbrowusers` с внешним ключом, каскадно удаляющим данные при удалении пользователя.

- **`setup/xtradbrowusers.install.php`**  
  Добавляет 15 демонстрационных полей (всех типов, поддерживаемых Cotonti Extrafields) и создаёт папку для загружаемых файлов.

- **`setup/xtradbrowusers.uninstall.sql`**  
  Удаляет таблицу `cot_xtradbrowusers` при деинсталляции.

- **`setup/xtradbrowusers.uninstall.php`**  
  Удаляет все определения экстраполей из `cot_extra_fields`, удаляет таблицу и папку с загруженными файлами, обеспечивая полную очистку системы.

#### 3. Интеграция с интерфейсом управления экстраполями

- **`xtradbrowusers.extrafields.php`**  
  Хук: `admin.extrafields.first`. Регистрирует таблицу `cot_xtradbrowusers` в списке доступных для управления экстраполями. Благодаря этому файлу администратор может добавлять, редактировать и удалять поля через стандартный раздел «Администрирование → Расширения → Экстраполя». В массиве `$extra_whitelist` задаются подсказки по доступным тегам для каждого шаблона.

#### 4. Редактирование профиля самим пользователем

- **`xtradbrowusers.users.profile.tags.php`**  
  Хук: `users.profile.tags`. Назначает теги `{USERS_PROFILE_XTRA_ИМЯПОЛЯ}` и `{USERS_PROFILE_XTRA_ИМЯПОЛЯ_TITLE}` в шаблоне редактирования профиля (`users.profile.tpl`). Позволяет как выводить поля в цикле (`<!-- BEGIN: XTRA_EXTRAFLD -->`), так и по отдельности.

- **`xtradbrowusers.users.profile.update.done.php`**  
  Хук: `users.profile.update.done`. После сохранения профиля пользователем перехватывает значения полей из POST, валидирует их через API и сохраняет в таблицу `cot_xtradbrowusers`, включая загрузку файлов.

#### 5. Редактирование пользователя администратором

- **`xtradbrowusers.users.edit.tags.php`**  
  Хук: `users.edit.tags`. Назначает теги `{USERS_EDIT_XTRA_ИМЯПОЛЯ}` и `{USERS_EDIT_XTRA_ИМЯПОЛЯ_TITLE}` в админской форме редактирования (`users.edit.tpl`).

- **`xtradbrowusers.users.edit.update.done.php`**  
  Хук: `users.edit.update.done`. Сохраняет данные после редактирования пользователя администратором. Аналогичен `users.profile.update.done`, но обрабатывает случай `$id` из запроса.

#### 6. Публичный профиль пользователя

- **`xtradbrowusers.users.details.tags.php`**  
  Хук: `users.details.tags`. Выводит все поля на публичной странице профиля (`users.details.tpl`). Теги имеют префикс `{USERS_DETAILS_XTRA_...}`. Поддерживается как цикл, так и индивидуальные теги. Для полей типа `country` дополнительно присваивается тег с названием страны (`_NAME`).

#### 7. Список пользователей

- **`xtradbrowusers.users.loop.php`**  
  Хук: `users.loop`. Внутри цикла вывода пользователей (например, в `users.tpl`) назначает теги `{USERS_ROW_XTRA_...}`. Работает для каждой строки списка.

- **`xtradbrowusers.usertags.php`**  
  Хук: `usertags.main`. Интегрируется в функцию `cot_generate_usertags()`. Добавляет в массив `$temp_array` теги с префиксом `XTRA_`, которые затем могут использоваться в любом контексте, где вызывается эта функция (форумы, списки, карточки и т.д.). Благодаря этому поля плагина доступны везде, где используются пользовательские теги, с единым синтаксисом.

#### 8. Регистрация новых пользователей

- **`xtradbrowusers.users.register.tags.php`**  
  Хук: `users.register.tags`. Назначает теги `{USERS_REGISTER_XTRA_...}` в форме регистрации (`users.register.tpl`).

- **`xtradbrowusers.users.register.add.done.php`**  
  Хук: `users.register.add.done`. Сохраняет значения дополнительных полей после успешной регистрации.

#### 9. Удаление пользователя

- **`xtradbrowusers.users.delete.done.php`**  
  Хук: `users.delete.done`. При удалении пользователя администратором или через валидацию регистрации сначала удаляет загруженные файлы (вызывая `cot_extrafield_unlinkfiles`), затем удаляет соответствующую запись из `cot_xtradbrowusers`. Гарантирует отсутствие мусора.

#### 10. Дополнительные интеграции

- **`xtradbrowusers.header.tags.php`**  
  Хук: `header.tags`. При просмотре профиля пользователя (когда определена переменная `$urr`) добавляет теги `{USERS_HEADER_XTRA_...}`, позволяя вывести, например, название страны или уровень квалификации в `<title>` или мета-тегах страницы.

#### API-функции плагина

- **`inc/xtradbrowusers.functions.php`**  
  Содержит три ключевые функции, используемые всеми файлами плагина:
  - `xtradbrowusers_getExtrafields()` — возвращает массив зарегистрированных экстраполей для таблицы `cot_xtradbrowusers`.
  - `xtradbrowusers_load($user_id)` — загружает данные дополнительных полей конкретного пользователя.
  - `xtradbrowusers_save($user_id, $data)` — сохраняет или обновляет запись в таблице, автоматически определяя, существует ли она.

Этот файл также регистрирует таблицу `xtradbrowusers` в системе и подключает языковые файлы.

## Часть №2: Файлы плагина и их роль

Плагин состоит из нескольких групп файлов, каждый из которых отвечает за свою часть работы с дополнительными полями пользователей. Ниже по группам описано назначение и механизм работы каждого файла.


---

### 1. Файлы регистрации и инициализации плагина

#### Файл `xtradbrowusers.setup.php`

Это **файл метаданных плагина**, который не является обработчиком какого‑либо хука. Он содержит единственный блок `[BEGIN_COT_EXT]...[END_COT_EXT]`, в котором через пары `Ключ=Значение` объявлены основные сведения о расширении:

- `Code` – уникальный идентификатор плагина (`xtradbrowusers`);
- `Name` – человекочитаемое название (`Custom Extrafields Users`);
- `Description`, `Version`, `Date`, `Author`, `Copyright`, `Notes` – справочная информация;
- `Requires_modules` – список обязательных модулей (в нашем случае `users`);
- `Auth_guests`, `Lock_guests`, `Auth_members`, `Lock_members` – права доступа для гостей и участников, которые будут записаны в таблицу `cot_auth` при установке.

**Этот файл не содержит PHP‑кода, исполняемого при каждом запросе.** Он обрабатывается **единожды** функцией `cot_extension_install()` во время установки плагина. При этом система Cotonti автоматически:

- создаёт запись в таблице `cot_core` с `ct_code='xtradbrowusers'`, `ct_plug=1` (признак плагина), текущей версией;
- на основе значений `Auth_guests`, `Lock_guests`, `Auth_members`, `Lock_members` формирует строки в таблице `cot_auth` для каждой группы пользователей (гости, неактивные, участники, суперадминистраторы и т.д.);
- сканирует все файлы плагина, находит в них заголовки вида `Hooks=...`, и для каждого создаёт отдельную запись в таблице `cot_plugins` с указанием хука, части, порядка выполнения и пути к файлу.

Сам `setup.php` **не вызывает** `Cot::$db->registerTable('xtradbrowusers')` и не подключает языковые файлы — эта инициализация выполняется позже, в рабочих файлах плагина.  

Файл не добавляет записей в `cot_config`, так как плагин пока не имеет собственных настроек, однако при необходимости разработчик может расширить `setup.php` блоком `COT_EXT_CONFIG`, и система автоматически создаст нужные опции в таблице `cot_config` (структура `cot_config` полностью позволяет хранить любые параметры модулей и плагинов).

#### Файл `xtradbrowusers.global.php`

Этот файл уже содержит **активный хук** `global`. Благодаря записи в `cot_plugins` он подключается **при каждом запросе** к сайту (до формирования основного вывода). Его задача — гарантировать, что во всех точках выполнения Cotonti доступны:

- языковые константы плагина (через `cot_langfile('xtradbrowusers', 'plug')`);
- API‑функции плагина, определённые в `inc/xtradbrowusers.functions.php`.

Именно внутри `xtradbrowusers.functions.php` (подключаемого отсюда) находится вызов `Cot::$db->registerTable('xtradbrowusers')`, регистрирующий глобальную переменную `$db_xtradbrowusers` и делающий доступным обращение `Cot::$db->xtradbrowusers` во всём коде плагина и за его пределами.  

Таким образом, `global.php` обеспечивает «фоновую» инициализацию, не требующую повторения в каждом хуке, а `setup.php` выполняет разовую регистрацию плагина в системных таблицах при установке.


### 2. Файлы установки и удаления (папка `setup/`)

#### Файл `setup/xtradbrowusers.install.sql`

SQL-скрипт, выполняемый однократно при установке плагина. Создаёт таблицу `cot_xtradbrowusers` (точное имя зависит от префикса БД) со следующей структурой:

- `user_id` — первичный ключ, целое беззнаковое, связанное через внешний ключ `fk_xtradbrowusers_users` с полем `user_id` таблицы `cot_users` и настроенным каскадным удалением `ON DELETE CASCADE`. Это означает, что при удалении пользователя из `cot_users` запись в `cot_xtradbrowusers` будет удалена автоматически на уровне БД, даже если код плагина по какой-то причине не отработает. Никаких дополнительных столбцов в этой таблице нет — все колонки будут добавлены позже через API Extrafields, когда администратор создаст поля в админке.

#### Файл `setup/xtradbrowusers.install.php`

Выполняется на этапе установки после SQL-файла. Его задача — заполнить свежесозданную таблицу наборами демонстрационных экстраполей, используя штатную функцию Cotonti `cot_extrafield_add()`. Для таблицы `cot_xtradbrowusers` добавляется 15 полей всех основных типов: `input`, `textarea`, `datetime`, `double`, `select`, `inputint`, `radio`, `file`, `country`, `range`, `checklistbox`. Каждое поле получает осмысленное название, HTML-шаблон для формы ввода, значения по умолчанию и описание. Благодаря этому администратор сразу после установки видит готовые примеры в панели управления экстраполями и в профилях пользователей.

Также создаётся папка `datas/exflds/xtradbrowusers` (если её нет) с правами 0777, куда будут сохраняться загружаемые через поля файлы. При необходимости можно раскомментировать создание `.htaccess` для запрета прямого доступа.

#### Файл `setup/xtradbrowusers.uninstall.sql`

Содержит единственный запрос `DROP TABLE IF EXISTS ...`, который удаляет таблицу `cot_xtradbrowusers` при деинсталляции плагина. Выполняется после PHP-кода удаления.

#### Файл `setup/xtradbrowusers.uninstall.php`

Этот файл вызывается при удалении плагина через панель администратора. Он производит полную очистку:

1. Удаляет все строки из `cot_extra_fields`, где `field_location` равно имени нашей таблицы — тем самым убираются все определения экстраполей из реестра.
2. Явно удаляет таблицу `cot_xtradbrowusers` через `DROP TABLE IF EXISTS` (на случай, если SQL-файл не отработал или префикс отличается).
3. Рекурсивно удаляет директорию `datas/exflds/xtradbrowusers` со всеми загруженными файлами, чтобы не оставлять файлового мусора. Используется простой рекурсивный обход папки.

### 3. Интеграция с управлением экстраполями в админке

Хук: `admin.extrafields.first`.


При установке плагина система Cotonti обнаруживает этот заголовок и добавляет в таблицу `cot_plugins` запись с `pl_hook='admin.extrafields.first'`, `pl_code='xtradbrowusers'`, `pl_part='extrafields'`, `pl_file='xtradbrowusers/xtradbrowusers.extrafields.php'`. - `'name'` — имя таблицы,
- `'caption'` — заголовок из языкового файла,
- `'type' => 'plug'` — признак плагина,
- `'code' => 'xtradbrowusers'` — идентификатор плагина,
- `'tags'` — массив подсказок для шаблонов: какие теги будут доступны в каждом файле темы (`users.profile.tpl`, `users.edit.tpl`, `users.details.tpl`, `users.register.tpl`, `users.tpl`, `header.tpl`).

Благодаря этому администратор видит нашу таблицу в выпадающем списке и может управлять её полями через стандартный интерфейс.

### 11. API-функции плагина

#### Файл `inc/xtradbrowusers.functions.php`

Содержит три функции, образующие ядро логики плагина:

- **`xtradbrowusers_getExtrafields()`** — возвращает массив зарегистрированных экстраполей для таблицы `cot_xtradbrowusers` из глобального реестра `Cot::$extrafields`. Если таблица ещё не инициализирована, возвращает пустой массив.
- **`xtradbrowusers_load($user_id)`** — выполняет `SELECT * FROM ... WHERE user_id = ?` и возвращает ассоциативный массив всех значений дополнительных полей для заданного пользователя или `null`, если записи нет.
- **`xtradbrowusers_save($user_id, $data)`** — проверяет, существует ли уже запись для этого пользователя, и либо обновляет её через `UPDATE`, либо вставляет новую через `INSERT`. При вставке автоматически добавляет ключ `user_id`.

Также этот файл подключает языковой файл плагина, файл API Extrafields и выполняет `Cot::$db->registerTable('xtradbrowusers')`, обеспечивая доступ к таблице по `Cot::$db->xtradbrowusers` в любой точке системы, где вызваны функции плагина.

## Часть №3: Управление экстраполями в админке — руководство для начинающих

После установки плагина вы попадаете в знакомую среду управления дополнительными полями Cotonti. Все операции выполняются через раздел **«Администрирование → Расширения → Экстраполя»**. В выпадающем списке таблиц нужно выбрать `cot_xtradbrowusers — Custom Extrafields Users`. Именно здесь создаются, изменяются и удаляются поля, которые затем появятся в профилях пользователей.

Интерфейс состоит из списка уже существующих полей (сразу после установки их 15) и формы добавления нового поля внизу страницы. Каждое поле характеризуется набором атрибутов, которые определяют его поведение и внешний вид. Давайте разберём их по порядку.

### Атрибуты экстраполя

- **Имя** — уникальное системное название поля (только латиница, цифры и знак подчёркивания). Например, `phone_extra` или `hire_date`. Именно это имя будет использоваться в шаблонах для вывода значения.
- **Тип** — определяет, какой HTML-элемент будет отображаться в форме редактирования и как данные будут обрабатываться. Доступны следующие типы:
  - `input` — обычное текстовое поле.
  - `inputint` — текстовое поле для целых чисел.
  - `currency`, `double` — текстовое поле для чисел с плавающей точкой.
  - `textarea` — многострочная область ввода.
  - `select` — выпадающий список.
  - `checkbox` — единичный флажок (да/нет).
  - `radio` — радиокнопки (выбор одного варианта из нескольких).
  - `datetime` — набор выпадающих списков для выбора даты и времени.
  - `country` — выпадающий список стран.
  - `range` — выпадающий список чисел из заданного диапазона.
  - `checklistbox` — группа чекбоксов для множественного выбора.
  - `file` — поле для загрузки файла.
- **HTML-код** — шаблон, который используется при отображении поля в форме. Обычно содержит переменные `{$name}`, `{$value}` и т.д. Если оставить поле пустым, система подставит стандартный HTML для выбранного типа.
- **Варианты** — строка с перечислением вариантов, разделённых запятыми. Используется для типов `select`, `radio`, `checklistbox`, а также для `file` (список разрешённых расширений). Например: `IT,Маркетинг,Продажи` или `Yes,No`.
- **Параметры** — дополнительные настройки поля. Для `datetime` это диапазон лет и формат, например `2010,2030,d.m.Y H:i`. Для `range` — минимальное и максимальное значение, например `0,100`. Для `double` и `inputint` можно задать границы через запятую (`min,max`).
- **Значение по умолчанию** — будет подставлено, если пользователь ничего не ввёл. Для `datetime` указывается timestamp (например `1714060800` для 26.04.2024), для остальных типов — просто строка или число.
- **Обязательное** — галочка, которая заставляет систему проверять заполнение поля перед сохранением.
- **Парсер** — определяет, как значение будет обрабатываться при выводе. `HTML` — разрешены HTML‑теги (но с очисткой), `Text` — всё преобразуется в plain text. Для `file`, `country`, `range` парсер, как правило, не применяется.
- **Описание** — человекочитаемое название поля, которое показывается рядом с полем в формах и в списке экстраполей. Именно оно становится заголовком `_TITLE` в шаблонах.

### Пошаговый пример: создание нового поля «Любимый язык программирования»

1. Перейдите в раздел экстраполей для таблицы `cot_xtradbrowusers`.
2. В форме добавления нового поля заполните:
   - **Имя:** `fav_language`
   - **Тип:** `select` (выпадающий список)
   - **Варианты:** `Не указан,JavaScript,Python,PHP,Rust`
   - **Значение по умолчанию:** `Не указан`
   - **Обязательное:** можно не ставить
   - **Парсер:** `HTML`
   - **Описание:** `Любимый язык программирования`
   - Остальные поля оставьте пустыми.
3. Нажмите кнопку «Добавить поле».

Новое поле немедленно появится в списке, и в профилях пользователей станет доступен выпадающий список с указанными вариантами.

## Часть 4: Вывод полей в шаблонах — настройка шаблона `users.edit.tpl` (редактирование администратором)

Плагин автоматически делает дополнительные поля доступными в каждом шаблоне через систему хуков. После того как поля созданы в админке, их нужно «подключить» к шаблонам — то есть вставить в нужное место специальные TPL‑теги. Начнём с формы редактирования пользователя администратором (`users.edit.tpl`).

### Файл плагина: `xtradbrowusers.users.edit.tags.php`

Этот файл зарегистрирован на хук `users.edit.tags` и выполняется каждый раз, когда Cotonti обрабатывает шаблон `users.edit.tpl`. Он загружает текущие значения дополнительных полей пользователя из таблицы `cot_xtradbrowusers` и присваивает их шаблонным переменным. Доступны два режима вывода: **циклический** (все поля разом) и **индивидуальный** (каждое поле отдельно).

### Какие теги доступны в шаблоне

После обработки хука в шаблоне можно использовать:

| Шаблонный тег | Назначение |
|---------------|------------|
| `{USERS_EDIT_XTRA_ИМЯПОЛЯ}` | HTML‑элемент формы (input, select, textarea…) с текущим значением |
| `{USERS_EDIT_XTRA_ИМЯПОЛЯ_TITLE}` | Название поля (подпись) |
| `{USERS_EDIT_XTRA_EXTRAFLD}` | Универсальный контейнер текущего поля (внутри цикла) |
| `{USERS_EDIT_XTRA_EXTRAFLD_TITLE}` | Название поля внутри цикла |

Здесь `ИМЯПОЛЯ` — это системное имя, которое вы задали при создании экстраполя (например, `phone_extra`), приведённое к верхнему регистру.

### Как вставить поля в шаблон

#### 1. Циклический вывод всех полей

Добавьте в шаблон следующий блок в том месте, где должны появиться дополнительные поля:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card border-0 shadow-sm mb-4">
    <div class="card-header">
        <h4>{PHP.L.xtradbrowusers_edit_tpl_title}</h4>
    </div>
    <div class="card-body">
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <div class="mb-3">
            <label class="form-label fw-semibold">{USERS_EDIT_XTRA_EXTRAFLD_TITLE}:</label>
            <div>{USERS_EDIT_XTRA_EXTRAFLD}</div>
        </div>
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

Этот код:
- проверяет, что плагин `xtradbrowusers` активен;
- выводит все поля одно за другим внутри цикла `<!-- BEGIN: XTRA_EXTRAFLD -->…<!-- END: XTRA_EXTRAFLD -->`;
- для каждого поля показывает его название и сам элемент формы.

Такой подход удобен, если вы хотите быстро отобразить все существующие дополнительные поля.

#### 2. Индивидуальный вывод конкретного поля

Если нужно расположить поля в разных местах карточки или применить уникальную вёрстку, используйте персональные теги. Например, поле с именем `phone_extra` будет доступно как `{USERS_EDIT_XTRA_PHONE_EXTRA}` и `{USERS_EDIT_XTRA_PHONE_EXTRA_TITLE}`.

Пример вставки:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<!-- IF {USERS_EDIT_XTRA_PHONE_EXTRA} -->
<div class="row mb-3">
    <label class="col-sm-3 col-form-label fw-semibold">{USERS_EDIT_XTRA_PHONE_EXTRA_TITLE}:</label>
    <div class="col-sm-9">{USERS_EDIT_XTRA_PHONE_EXTRA}</div>
</div>
<!-- ENDIF -->
<!-- ENDIF -->
```

Таким образом можно комбинировать произвольное количество полей, создавая сложную разметку.

#### 3. Пример для вашего шаблона `users.edit.tpl` (тема Index36)

Допустим, мы хотим разместить дополнительные поля плагина между блоком «Основные данные» и «Подвал с подписью». В вашем шаблоне после закрывающего тега `</div>` блока с группами и перед строкой с `<!-- IF {PHP|cot_module_active('pm')} -->` можно вставить:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card border-0 shadow-sm mb-4">
    <div class="card-header">
        <h4>{PHP.L.xtradbrowusers_edit_tpl_title}</h4>
    </div>
    <div class="card-body">
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <div class="mb-3">
            <label class="form-label fw-semibold">{USERS_EDIT_XTRA_EXTRAFLD_TITLE}:</label>
            <div>{USERS_EDIT_XTRA_EXTRAFLD}</div>
        </div>
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

Либо вывести отдельные поля в нужных вам местах, используя индивидуальные теги.

### Проверка и сохранение

После вставки кода в шаблон откройте профиль любого пользователя для редактирования администратором (например, `/users/1?m=edit`). Вы должны увидеть новые поля. Заполните их, сохраните — данные запишутся в таблицу `cot_xtradbrowusers` и будут доступны для других частей системы (публичный профиль, список пользователей и т.д.).

В следующей части мы аналогичным образом настроим публичный профиль пользователя (`users.details.tpl`) и другие шаблоны.

## Часть 6: Вывод полей в публичном профиле пользователя (`users.details.tpl`)

Публичная страница профиля пользователя (`/users/...?m=details`) — это то место, где посетители сайта видят информацию о пользователе. Плагин `xtradbrowusers` автоматически делает дополнительные поля доступными и здесь через хук `users.details.tags`, который обрабатывается файлом `xtradbrowusers.users.details.tags.php`. В отличие от форм редактирования, здесь поля отображаются **не как элементы ввода**, а как готовые отформатированные значения, подходящие для чтения.

### Какие теги доступны в шаблоне

После обработки хука в `users.details.tpl` можно использовать следующие теги:

| Шаблонный тег | Назначение |
|---------------|------------|
| `{USERS_DETAILS_XTRA_ИМЯПОЛЯ}` | Отформатированное значение поля (текст, HTML) |
| `{USERS_DETAILS_XTRA_ИМЯПОЛЯ_TITLE}` | Название поля (подпись) |
| `{USERS_DETAILS_XTRA_ИМЯПОЛЯ_VALUE}` | Сырое значение поля (без форматирования, может быть полезно для сравнений) |
| `{USERS_DETAILS_XTRA_EXTRAFIELD_TITLE}` | Название текущего поля (внутри цикла) |
| `{USERS_DETAILS_XTRA_EXTRAFIELD_VALUE}` | Отформатированное значение текущего поля (внутри цикла) |
| `{USERS_DETAILS_XTRA_EXTRAFIELD_NAME}` | Системное имя текущего поля (внутри цикла) |

Имена полей пишутся в верхнем регистре, например для поля `phone_extra` — `USERS_DETAILS_XTRA_PHONE_EXTRA`.

## Часть 7: Вывод полей в общем списке пользователей (`users.tpl`)

| Шаблонный тег | Назначение |

| `{USERS_ROW_XTRA_ИМЯПОЛЯ}` | Отформатированное значение поля |
| `{USERS_ROW_XTRA_ИМЯПОЛЯ_TITLE}` | Название поля (подпись) |
| `{USERS_ROW_XTRA_ИМЯПОЛЯ_VALUE}` | Сырое значение поля (без форматирования) |

Для полей типа `country` дополнительно:

| `{USERS_ROW_XTRA_ИМЯПОЛЯ_NAME}` | Полное название страны |

Имена полей записываются в верхнем регистре с префиксом `XTRA_`, например для поля `phone_extra` — `USERS_ROW_XTRA_PHONE_EXTRA`. Обратите внимание: в `users.details.tpl` использовался префикс `USERS_DETAILS_XTRA_`, здесь же префикс `USERS_ROW_XTRA_`.

## Часть 8: Вывод полей в форме регистрации (`users.register.tpl`)

Форма регистрации нового пользователя — это первая точка входа, где можно собрать дополнительную информацию о будущем участнике. Плагин `xtradbrowusers` интегрируется в эту форму через файл `xtradbrowusers.users.register.tags.php`, зарегистрированный на хук `users.register.tags`.

Поскольку в момент отображения формы пользователь ещё не создан (его нет в базе), плагин не загружает данные из таблицы `cot_xtradbrowusers`, а ориентируется на массив `$ruser`, который содержит значения, введённые при предыдущей попытке отправки (если были ошибки валидации). Это позволяет сохранить введённые данные в полях после перезагрузки страницы, чтобы пользователю не приходилось заполнять всё заново.

### Какие теги доступны в шаблоне

После обработки хука в шаблоне `users.register.tpl` можно использовать:

| Шаблонный тег | Назначение |
|---------------|------------|
| `{USERS_REGISTER_XTRA_ИМЯПОЛЯ}` | HTML‑элемент формы (input, select, textarea…) с пустым или сохранённым значением |
| `{USERS_REGISTER_XTRA_ИМЯПОЛЯ_TITLE}` | Название поля (подпись) |
| `{USERS_REGISTER_XTRA_EXTRAFLD}` | Универсальный контейнер текущего поля (внутри цикла) |
| `{USERS_REGISTER_XTRA_EXTRAFLD_TITLE}` | Название поля внутри цикла |

Имена полей записываются в верхнем регистре, например для поля `phone_extra` — `USERS_REGISTER_XTRA_PHONE_EXTRA`.

### Как вставить поля в шаблон

Обязательно оборачивайте вывод в проверку активности плагина, чтобы при его отключении вёрстка осталась корректной:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
...
<!-- ENDIF -->
```

#### 1. Циклический вывод всех полей

Можно вывести все дополнительные поля единым блоком, используя цикл. Это удобно, если вы не хотите настраивать каждое поле отдельно:

```html

    <!-- BEGIN: XTRA_EXTRAFLD -->

        {USERS_REGISTER_XTRA_EXTRAFLD}
    </div>
    <!-- END: XTRA_EXTRAFLD -->
</div>
<!-- ENDIF -->
```

#### 2. Индивидуальный вывод отдельных полей

Если вы хотите расположить определённые поля в разных местах формы или оформить их по-особому, используйте персональные теги. Например, поле «Дополнительный телефон» (`phone_extra`):

## Часть 9: Локализация дополнительных полей

Плагин `xtradbrowusers` полностью совместим с системой локализации Cotonti. Это значит, что названия полей и их значения (например, варианты в выпадающих списках или радиокнопках) можно переводить на разные языки, и они будут автоматически отображаться в соответствии с выбранным языком сайта. Вам не нужно дублировать поля для каждой языковой версии — достаточно один раз создать поле в админке и добавить переводы в языковые файлы плагина.

### 1. Где хранятся переводы

Языковые файлы плагина находятся в папке `lang/`:

- `xtradbrowusers.ru.lang.php` — русский перевод
- `xtradbrowusers.en.lang.php` — английский перевод

Вы можете редактировать эти файлы или создавать новые для других языков (например, `xtradbrowusers.ua.lang.php` для украинского). Также переводы можно размещать в языковых файлах вашей темы — они будут иметь приоритет над файлами плагина.

### 2. Перевод названий полей (заголовков)

Название поля (то, что выводится рядом с полем в форме, и что доступно в шаблоне как `_TITLE`) определяется функцией `cot_extrafield_title()`, которая ищет языковую строку в следующем порядке:

1. `$L['xtra_имя_поля_title']` — с префиксом `xtra_` (этот префикс плагин передаёт при вызове)
2. `$L['имя_поля_title']` — без префикса
3. Если ничего не найдено, используется значение из поля «Описание» (`field_description`) в админке.

Например, для поля `phone_extra` система проверит:

- `$L['xtra_phone_extra_title']`
- `$L['phone_extra_title']`

Если ни один ключ не найден, подставится описание из админки «Дополнительный телефон».

#### Пример: перевод названия поля «Дополнительный телефон» на русский и английский

Добавьте в `lang/xtradbrowusers.ru.lang.php`:

```php
$L['xtra_phone_extra_title'] = 'Дополнительный телефон';
```

Или без префикса:

```php
$L['phone_extra_title'] = 'Дополнительный телефон';
```

В `lang/xtradbrowusers.en.lang.php`:

```php
$L['xtra_phone_extra_title'] = 'Extra phone';
```

Теперь в русской версии сайта рядом с полем будет написано «Дополнительный телефон», а в английской — «Extra phone».

### 3. Перевод значений для выпадающих списков, радиокнопок и чекбоксов

Для типов полей `select`, `radio`, `checklistbox` можно локализовать не только название, но и **варианты выбора**. В базе данных значения хранятся в исходном виде (например, `it_department`), а при выводе функция `cot_build_extrafields_data()` ищет перевод по ключу `$L['имя_поля_значение']` (в нижнем регистре). Префиксы здесь не используются.

Формат ключа: `$L['fieldname_value']`  
Пример для поля `department` со значением `it_department`:

```php
$L['department_it_department'] = 'IT-отдел';
```

Если перевода нет, отображается исходное значение.

#### Пример: локализация выпадающего списка «Отдел» (department)

Поле `department` в админке имеет варианты: `Не указан,IT,Маркетинг,Продажи,Поддержка` (это значения по-русски, но технически они могут быть любыми строками). Чтобы они переводились при смене языка, в языковые файлы добавляют:

**Русский (`ru.lang.php`):**
```php
$L['department_Не указан'] = 'Не указан';
$L['department_IT'] = 'IT-отдел';
$L['department_Маркетинг'] = 'Маркетинг';
$L['department_Продажи'] = 'Продажи';
$L['department_Поддержка'] = 'Поддержка';
```

**Английский (`en.lang.php`):**
```php
$L['department_Не указан'] = 'Not specified';
$L['department_IT'] = 'IT Department';
$L['department_Маркетинг'] = 'Marketing';
$L['department_Продажи'] = 'Sales';
$L['department_Поддержка'] = 'Support';
```

Теперь при выводе значения поля (например, `{USERS_DETAILS_XTRA_DEPARTMENT}`) вместо «IT» английский пользователь увидит «IT Department».

### 4. Перевод значений для `radio` и `checklistbox`

Работает точно так же: ключ `$L['fieldname_value']`. Например, для поля `has_car` со значениями `Yes,No`:

```php
// Русский
$L['has_car_Yes'] = 'Да';
$L['has_car_No'] = 'Нет';

// Английский
$L['has_car_Yes'] = 'Yes';
$L['has_car_No'] = 'No';
```

### 6. Приоритет языковых файлов

Cotonti загружает языковые файлы в следующем порядке:

1. Глобальные языковые файлы (`lang/ru/main.ru.lang.php`)
2. Языковые файлы модулей (`modules/*/lang/ru/ru.*.lang.php`)
3. Языковые файлы плагинов (`plugins/*/lang/ru/ru.*.lang.php`) — здесь находятся файлы нашего плагина
4. Языковой файл темы (`themes/ваша_тема/ваша_тема.ru.lang.php`) — загружается последним и может переопределить любые строки

Поэтому, если вы хотите изменить перевод, не трогая файлы плагина, просто добавьте нужные ключи в языковой файл вашей темы. Например, в `themes/index36/index36.ru.lang.php`:

```php
$L['department_IT'] = 'Отдел информационных технологий';
```

Это значение переопределит то, что задано в файле плагина.

### 7. Рекомендация для начинающих

- Всегда задавайте в админке варианты значений на том языке, который вы будете использовать в качестве основного (например, русский). Затем добавляйте переводы для других языков через `$L['fieldname_value']`.  
- Если не планируется многоязычность, можно просто заполнить варианты сразу на нужном языке и не добавлять переводы — тогда будет выводиться то, что указано в админке.  
- Для заголовков полей, если вас устраивает текст из поля «Описание», дополнительные языковые строки можно не создавать. Однако при смене языка описание не переведётся автоматически, поэтому лучше использовать ключи локализации.

Теперь ваш плагин готов к использованию на многоязычном сайте — все заголовки и значения дополнительных полей могут быть переведены на любой язык без вмешательства в код.
Всё происходит автоматически — именно для этого плагин использует штатные функции Cotonti. Когда вы вставляете в шаблон тег вроде `{USERS_DETAILS_XTRA_DEPARTMENT}`, он уже содержит **готовую локализованную строку**.  
Вам **не нужно** писать в шаблоне никаких дополнительных условий или вызовов языковых файлов.


### Как это работает «под капотом»

1. **Значения полей**  
   В хуках вывода (например, `users.details.tags`) плагин вызывает `cot_build_extrafields_data('xtra', $exfld, $value)`.  
   Эта функция, обнаружив тип `select`, `radio` или `checklistbox`, автоматически ищет в массиве `$L` ключ `$L['имя_поля_значение']` и, если находит, подставляет перевод вместо исходного значения.  
   В шаблон попадает уже переведённый текст, поэтому никаких дополнительных действий не требуется.

2. **Заголовки полей**  
   Тег `_TITLE` (например, `{USERS_DETAILS_XTRA_DEPARTMENT_TITLE}`) формируется функцией `cot_extrafield_title($exfld, 'xtra_')`.  
   Она ищет ключи `$L['xtra_имя_поля_title']` и `$L['имя_поля_title']`. Если перевод найден — возвращается он, если нет — описание из админки.  
   Результат также сразу попадает в шаблон.

### Пример из вашего шаблона `users.details.tpl`

Допустим, у вас есть поле `department` со значением `IT`.  
В языковой файл добавлено:
```php
$L['department_IT'] = 'IT-отдел';
$L['xtra_department_title'] = 'Отдел';
```

В шаблоне вы пишете:
```html
<!-- IF {USERS_DETAILS_XTRA_DEPARTMENT} -->
<div class="mb-3">
    <strong>{USERS_DETAILS_XTRA_DEPARTMENT_TITLE}:</strong>
    <span>{USERS_DETAILS_XTRA_DEPARTMENT}</span>
</div>
<!-- ENDIF -->
```

Пользователь увидит:  
**Отдел:** IT-отдел  

Если он переключит язык на английский, и в `en.lang.php` будет:
```php
$L['department_IT'] = 'IT Department';
$L['xtra_department_title'] = 'Department';
```
То вывод автоматически изменится на:  
**Department:** IT Department  

### Что нужно помнить

- **Никаких ручных вызовов `cot_langfile()` в шаблоне** — языковой файл плагина уже загружен глобально (через `xtradbrowusers.global.php`).  
- **Если вы добавляете переводы в тему**, они подхватятся автоматически, потому что файл темы загружается позже и дополняет/переопределяет массив `$L`.  
- **Значения, для которых нет перевода**, отображаются «как есть» (например, `IT`). Это нормально и позволяет постепенно добавлять локализацию.  

Таким образом, **вся локализация уже встроена в теги**, и шаблоны остаются чистыми и не зависят от конкретного языка.

## Пошаговая установка плагина

### Шаг 1: Получение файлов плагина

Скачайте последнюю версию плагина `xtradbrowusers` из [официального репозитория GitHub](https://github.com/webitproff/xtradbrowusers-cotonti/releases). На странице релизов выберите архив `xtradbrowusers.zip` и сохраните его на свой компьютер.

### Шаг 2: Загрузка на сервер

Распакуйте архив — внутри находится папка `xtradbrowusers` со всеми файлами. Загрузите её в директорию `plugins/` вашего сайта на Cotonti.  
Проверьте, что файл `xtradbrowusers.setup.php` доступен по пути:  
`/plugins/xtradbrowusers/xtradbrowusers.setup.php`.  

Права доступа должны позволять веб-серверу читать файлы и создавать папки (стандартные `0755` для директорий и `0644` для файлов подходят).

### Шаг 3: Установка через панель администратора

1. Войдите в административную панель Cotonti.
2. Перейдите в раздел **«Расширения»** (обычно пункт «Extensions» или «Расширения» в главном меню админки). В зависимости от настроек сайта URL может выглядеть как `/admin/extensions`, `/ru/admin/extensions` или `admin.php?m=extensions`.
3. Найдите в списке плагинов **«Custom Extrafields Users»** (код `xtradbrowusers`). Если плагин не отображается, убедитесь, что файлы загружены в правильную папку и сервер их видит.
4. Нажмите кнопку **«Установить»** напротив названия плагина.

Система автоматически выполнит:

- Регистрацию плагина в таблице `cot_core`.
- Создание прав доступа в `cot_auth` для всех групп пользователей.
- Запуск `setup/xtradbrowusers.install.sql` — будет создана таблица `cot_xtradbrowusers` с внешним ключом к `cot_users`.
- Запуск `setup/xtradbrowusers.install.php` — добавится 15 демонстрационных полей и появится папка `datas/exflds/xtradbrowusers` для загружаемых файлов.
- Сканирование всех файлов плагина и регистрацию хуков в таблице `cot_plugins`.

После успешной установки вы увидите подтверждающее сообщение.

### Шаг 4: Проверка

После установки:

1. Зайдите в **«Администрирование → Расширения → Экстраполя»**.
2. В выпадающем списке таблиц выберите `cot_xtradbrowusers — Custom Extrafields Users`. Вы увидите 15 готовых к использованию полей.
3. Можете сразу открыть профиль любого пользователя для редактирования (например, `/users/1?m=edit`) — в форме появятся все добавленные поля.

Дополнительная настройка через `cot_config` не требуется, плагин работает сразу после установки.

## Часть 11: Групповой (циклический) вывод всех дополнительных полей

Когда количество экстраполей измеряется десятками, перечислять каждое поле в шаблоне по отдельности становится неудобно. Плагин `xtradbrowusers` позволяет вывести сразу все активные дополнительные поля одной компактной конструкцией — циклом `<!-- BEGIN: XTRA_EXTRAFLD -->`. Этот механизм работает во **всех шаблонах**, куда плагин уже интегрирован, и не требует настройки каждого поля.

### Как это работает

Каждый файл-обработчик хука (например, `users.profile.tags.php`) внутри цикла по всем экстраполям плагина назначает пару универсальных тегов:

- **`{..._XTRA_EXTRAFLD}`** — HTML-код текущего поля (input, select, textarea и т.п.)
- **`{..._XTRA_EXTRAFLD_TITLE}`** — локализованное название этого поля

Эти теги обновляются на каждой итерации, а шаблонный цикл `<!-- BEGIN: XTRA_EXTRAFLD -->` проходит по всем полям и выводит их последовательно. Префикс тегов зависит от контекста (шаблона):

| Шаблон | Префикс тегов |
|--------|---------------|
| `users.edit.tpl` | `USERS_EDIT_XTRA_` |
| `users.profile.tpl` | `USERS_PROFILE_XTRA_` |
| `users.details.tpl` | `USERS_DETAILS_XTRA_` |
| `users.register.tpl` | `USERS_REGISTER_XTRA_` |
| `users.tpl` (список) | `USERS_ROW_XTRA_` |

### Правила интеграции

1. **Проверка активности плагина** — обязательна, как и при индивидуальном выводе.
2. Цикл размещается в том месте шаблона, где вы хотите видеть все дополнительные поля подряд.
3. Внутри цикла можно использовать любую HTML-разметку, а также проверять, не пустое ли текущее поле: `<!-- IF {USERS_EDIT_XTRA_EXTRAFLD} -->`.
4. Если какое-то поле отключено или удалено администратором, оно просто перестаёт выводиться — править шаблоны не нужно.

### Примеры для каждого шаблона

#### 1. Редактирование администратором (`users.edit.tpl`)

Вставьте блок, например, перед кнопкой «Сохранить»:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card border-0 shadow-sm mb-4">
    <div class="card-header">
        <h4>{PHP.L.xtradbrowusers_edit_tpl_title}</h4>
    </div>
    <div class="card-body">
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <div class="mb-3">
            <label class="form-label fw-semibold">{USERS_EDIT_XTRA_EXTRAFLD_TITLE}:</label>
            <div>{USERS_EDIT_XTRA_EXTRAFLD}</div>
        </div>
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

**Результат:** все дополнительные поля пользователя отобразятся одно за другим внутри карточки. Администратор может заполнить/изменить значения и сохранить их.

#### 2. Редактирование профиля пользователем (`users.profile.tpl`)

Аналогичный код, но с префиксом `USERS_PROFILE`:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card border-0 shadow-sm mb-4">
    <div class="card-header">
        <h4>{PHP.L.xtradbrowusers_profile_tpl_title}</h4>
    </div>
    <div class="card-body">
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <div class="mb-3">
            <label class="form-label fw-semibold">{USERS_PROFILE_XTRA_EXTRAFLD_TITLE}:</label>
            <div>{USERS_PROFILE_XTRA_EXTRAFLD}</div>
        </div>
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

#### 3. Публичный профиль (`users.details.tpl`)

Здесь выводятся уже сохранённые значения (не формы ввода). Можно добавить проверку на пустоту, чтобы не показывать незаполненные поля:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card mb-4">
    <div class="card-header">
        <h4>{PHP.L.xtradbrowusers_details_tpl_title}</h4>
    </div>
    <div class="card-body">
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <!-- IF {USERS_DETAILS_XTRA_EXTRAFIELD_VALUE} -->
        <div class="mb-3">
            <strong>{USERS_DETAILS_XTRA_EXTRAFIELD_TITLE}:</strong>
            <span>{USERS_DETAILS_XTRA_EXTRAFIELD_VALUE}</span>
        </div>
        <!-- ENDIF -->
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

**Пояснение:** теги `USERS_DETAILS_XTRA_EXTRAFIELD_VALUE` и `USERS_DETAILS_XTRA_EXTRAFIELD_TITLE` созданы специально для страницы просмотра. Они содержат готовое значение поля и его название.

#### 4. Список пользователей (`users.tpl`)

В карточке каждого пользователя можно вывести все его дополнительные поля компактной строкой или списком. Важно: теги здесь имеют префикс `USERS_ROW_`, а цикл работает внутри блока `<!-- BEGIN: USERS_ROW -->`.

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="mt-2 small text-muted">
    <!-- BEGIN: XTRA_EXTRAFLD -->
    <!-- IF {USERS_ROW_XTRA_EXTRAFLD} -->
    <div>{USERS_ROW_XTRA_EXTRAFLD_TITLE}: {USERS_ROW_XTRA_EXTRAFLD}</div>
    <!-- ENDIF -->
    <!-- END: XTRA_EXTRAFLD -->
</div>
<!-- ENDIF -->
```

Такой блок размещают внутри карточки пользователя, сразу после основных данных (имени, группы).

#### 5. Форма регистрации (`users.register.tpl`)

Новые пользователи могут сразу заполнить дополнительные поля при регистрации:

```html
<!-- IF {PHP|cot_plugin_active('xtradbrowusers')} -->
<div class="card bg-light mb-4">
    <div class="card-body">
        <h6 class="card-title">{PHP.L.xtradbrowusers_register_title}</h6>
        <!-- BEGIN: XTRA_EXTRAFLD -->
        <div class="mb-3">
            <label class="form-label small">{USERS_REGISTER_XTRA_EXTRAFLD_TITLE}:</label>
            {USERS_REGISTER_XTRA_EXTRAFLD}
        </div>
        <!-- END: XTRA_EXTRAFLD -->
    </div>
</div>
<!-- ENDIF -->
```

Понял, исправляю текст с учётом реальной архитектуры плагина. В списке пользователей (`users.tpl`) у нас **нет** группового цикла — только индивидуальные теги `USERS_ROW_XTRA_...`. Поэтому рекомендации нужно скорректировать.

Вот исправленный фрагмент (заменяет раздел «Важно: нельзя смешивать…» и «Рекомендации по организации шаблонов»):

### Важно: нельзя смешивать индивидуальный и групповой вывод в одном шаблоне формы

В формах редактирования (`edit`, `profile`, `register`) **недопустимо одновременно использовать индивидуальный вывод конкретных полей и групповой цикл**. Причина техническая: если одно и то же поле (например, `interests`) выведено дважды, в HTML окажутся два элемента с одинаковым именем (`rxtra_interests[]`). При отправке формы браузер передаст значения из последнего встреченного элемента, что приведёт к потере данных или непредсказуемому результату.

**Для форм рекомендуется выбрать один способ:**
- **Групповой (цикл)** — идеален для стандартного отображения всех полей, когда не нужна особая вёрстка. Просто вставляете блок с `<!-- BEGIN: XTRA_EXTRAFLD -->`, и все поля выводятся автоматически.
- **Индивидуальный** — подходит, если требуется уникальное оформление определённых полей. В этом случае вы выводите каждое нужное поле индивидуальным тегом, а групповой блок **не используете**.

В публичных шаблонах, где нет отправки форм (`details.tpl`, `users.tpl`), дублирование не вызовет проблем с данными, но для читаемости кода его тоже следует избегать.

### Рекомендации по организации шаблонов

- **Формы редактирования (edit, profile, register):** если вы часто добавляете или удаляете поля в админке, используйте **только групповой цикл** — шаблон не придётся править. Если же вам нужна особая вёрстка для некоторых полей, переходите на **индивидуальный вывод** всех нужных полей (без цикла).

- **Публичная страница профиля (`users.details.tpl`):** здесь допустимы оба подхода. Цикл удобен, чтобы быстро отобразить все заполненные поля списком. Индивидуальный вывод незаменим, когда требуется кастомизация: флаги стран, прогресс-бары, ссылки на файлы. Для индивидуального вывода используйте теги `USERS_DETAILS_XTRA_...`.

- **Список пользователей (`users.tpl`):** в этом шаблоне плагин предоставляет **только индивидуальные теги** `USERS_ROW_XTRA_...` для каждого поля. Группового цикла здесь нет, потому что каждая карточка пользователя обычно требует особого оформления. Выводите нужные поля по отдельности.

- **Всегда проверяйте активность плагина** (`cot_plugin_active`) и (при желании) наличие значения, чтобы избежать пустых блоков. Код плагина не привязан к конкретной теме и будет работать в любой теме, использующей стандартные шаблоны Cotonti.
